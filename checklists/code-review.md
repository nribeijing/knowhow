# 代码审查清单

> 系统性审查代码，确保质量

**适用场景**: PR 审查、代码合并前  
**执行者**: 人类 (主审) + AI (辅助)  
**耗时**: 15-30 分钟

---

## 🎯 审查目标

1. **功能正确** - 代码按预期工作
2. **安全可靠** - 无安全漏洞
3. **性能合理** - 无明显性能问题
4. **易于维护** - 代码清晰易懂

---

## 📋 审查清单

### 1. 核心逻辑审查

#### 业务逻辑
- [ ] 需求是否完整实现
- [ ] 业务规则是否正确
- [ ] 边界条件是否处理
- [ ] 异常流程是否考虑

#### 数据流
- [ ] 数据来源是否可靠
- [ ] 数据处理是否正确
- [ ] 数据去向是否明确
- [ ] 数据验证是否完整

#### 状态管理
- [ ] 状态转换是否正确
- [ ] 状态持久化是否可靠
- [ ] 并发处理是否适当
- [ ] 死锁风险是否排除

---

### 2. 安全性审查

#### 认证授权
- [ ] 认证逻辑是否正确
- [ ] 授权检查是否到位
- [ ] 权限提升是否可能
- [ ] Session/Token 处理是否安全

#### 输入验证
- [ ] 所有输入是否验证
- [ ] 类型检查是否完整
- [ ] 长度限制是否适当
- [ ] 特殊字符是否处理

#### 数据安全
- [ ] SQL 注入风险
- [ ] XSS 攻击风险
- [ ] CSRF 攻击风险
- [ ] 敏感信息泄露风险

#### 依赖安全
- [ ] 依赖版本是否最新
- [ ] 已知漏洞是否修复
- [ ] 第三方库是否可信
- [ ] License 是否兼容

---

### 3. 性能审查

#### 数据库
- [ ] 查询是否优化
- [ ] 索引是否使用
- [ ] N+1 查询是否存在
- [ ] 连接池配置是否合理

#### 缓存
- [ ] 缓存策略是否适当
- [ ] 缓存命中率是否合理
- [ ] 缓存穿透/击穿/雪崩预防
- [ ] 缓存一致性是否保证

#### 算法复杂度
- [ ] 时间复杂度是否合理
- [ ] 空间复杂度是否合理
- [ ] 大 O 是否可接受
- [ ] 是否有更优算法

#### 资源使用
- [ ] 内存使用是否合理
- [ ] CPU 使用是否合理
- [ ] IO 操作是否优化
- [ ] 资源释放是否及时

---

### 4. 代码质量审查

#### 可读性
- [ ] 命名是否清晰
- [ ] 函数是否简短
- [ ] 注释是否适当
- [ ] 结构是否清晰

#### 可维护性
- [ ] 是否遵循 DRY
- [ ] 是否遵循 SRP
- [ ] 依赖是否合理
- [ ] 配置是否外置

#### 可测试性
- [ ] 是否易于测试
- [ ] 依赖是否可 Mock
- [ ] 副作用是否隔离
- [ ] 测试覆盖是否充分

#### 代码规范
- [ ] 是否遵循项目规范
- [ ] 是否遵循行业规范
- [ ] 风格是否一致
- [ ] Lint 是否通过

---

### 5. 测试审查

#### 测试覆盖
- [ ] 正常路径是否测试
- [ ] 异常路径是否测试
- [ ] 边界条件是否测试
- [ ] 并发场景是否测试

#### 测试质量
- [ ] 测试是否独立
- [ ] Mock 是否适当
- [ ] 断言是否明确
- [ ] 测试数据是否合理

#### 测试维护
- [ ] 测试是否稳定
- [ ] 测试是否易读
- [ ] 测试是否易改
- [ ] 测试文档是否完整

---

### 6. 文档审查

#### 代码文档
- [ ] 公共 API 是否有文档
- [ ] 复杂逻辑是否有注释
- [ ] 参数说明是否完整
- [ ] 返回值说明是否清晰

#### 项目文档
- [ ] README 是否更新
- [ ] CHANGELOG 是否更新
- [ ] API 文档是否更新
- [ ] 配置说明是否更新

#### 提交信息
- [ ] 提交信息是否规范
- [ ] 描述是否清晰
- [ ] 关联 Issue 是否正确
- [ ] 是否有 Breaking Change 说明

## 🔍 审查技巧

### 分层审查法

```
1. 架构层
   - 模块划分是否合理
   - 依赖关系是否清晰
   - 技术选型是否适当

2. 设计层
   - 设计模式是否适当
   - 接口设计是否合理
   - 扩展性是否考虑

3. 实现层
   - 代码是否正确
   - 性能是否合理
   - 安全是否有保障

4. 细节层
   - 命名是否清晰
   - 注释是否适当
   - 格式是否规范
```

### 重点审查项

**优先级排序**:
1. 🔴 **安全性** - 安全漏洞必须修复
2. 🔴 **正确性** - 功能错误必须修复
3. 🟡 **性能** - 性能问题评估影响
4. 🟢 **规范** - 规范问题建议修复

### 审查话术

**建设性反馈**:
```
❌ "这个实现有问题"
✅ "这里考虑过 XX 场景吗？建议..."

❌ "代码写得太乱了"
✅ "这段逻辑有点复杂，是否可以拆分成..."

❌ "性能太差了"
✅ "这个查询在大数据量下可能有性能问题，建议添加索引..."
```

## 📝 审查报告模板

```markdown
# 代码审查报告

**PR**: [#123] 功能描述
**审查人**: [姓名]
**审查日期**: 2026-03-15
**审查耗时**: 30 分钟

## 审查结论

- [ ] ✅ 批准合并
- [ ] ⚠️ 需要修改 (见下方)
- [ ] ❌ 需要重写 (见下方)

## 发现的问题

### 🔴 严重问题 (必须修复)

1. **问题**: [描述]
   **位置**: [文件：行号]
   **建议**: [修复方案]
   **原因**: [为什么是问题]

2. **问题**: [描述]
   ...

### 🟡 中等问题 (建议修复)

1. **问题**: [描述]
   **位置**: [文件：行号]
   **建议**: [改进方案]

### 🟢 轻微问题 (可选修复)

1. **问题**: [描述]
   **建议**: [改进建议]

## 优点

- [代码亮点 1]
- [代码亮点 2]
- [值得学习的地方]

## 后续跟进

- [ ] 问题已修复
- [ ] 已重新审查
- [ ] 批准合并
```

## 🚨 红旗警告 (必须拒绝)

以下情况 **必须拒绝合并**，直到修复：

- 🔴 安全漏洞未修复
- 🔴 核心功能错误
- 🔴 数据丢失风险
- 🔴 严重性能问题
- 🔴 测试覆盖不足 (<80%)
- 🔴 无测试代码

---

## 📊 审查指标

### 审查速度
- 小改动 (<100 行): 15 分钟
- 中改动 (100-500 行): 30 分钟
- 大改动 (>500 行): 60 分钟 + 分次审查

### 审查质量
- 问题发现率: >5 个/百行
- 严重问题遗漏: 0
- 审查满意度: >4/5

### 审查效率
- 一次通过率: >60%
- 平均修改轮次: <2 轮
- 平均合并时间: <24 小时

## 📈 持续改进

### 如何更新本清单

1. **发现问题** → 记录到清单
2. **解决问题** → 更新审查项
3. **定期回顾** → 删除过时项
4. **团队评审** → 统一标准

### 审查者成长路径

```
初级审查者:
- 能发现明显错误
- 关注代码规范
- 需要指导

中级审查者:
- 能发现潜在问题
- 关注架构设计
- 独立审查

高级审查者:
- 能发现系统性问题
- 关注安全性能
- 指导他人
```
