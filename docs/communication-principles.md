# 🤝 与 LLM 沟通的原则

> 核心原则：**清晰、完整、可验证**

---

## 一、清晰表达

### ✅ 好的表达

```
任务：修复 API 尾部斜杠问题

背景：
- FastAPI 对 API URL 的尾部斜杠很严格
- 缺少斜杠会导致 307 重定向
- 公网访问时会导致 Cookie 丢失

要求：
1. 检查后端所有 @router.* 定义
2. 生成完整的斜杠规则表
3. 更新前端拦截器
4. 全局搜索硬编码的斜杠
5. 测试所有端点

验收标准：
- 所有 API 请求 URL 符合后端路由定义
- 无 307 重定向
- Network 标签中请求 URL 正确
```

### ❌ 差的表达

```
这里有问题，修复一下
```

---

## 二、提供完整上下文

### ✅ 好的做法

```
后端路由规则如下：

列表端点（需要 /）：
- @router.get("/") → /roles/, /users/, /museums/

单个资源（不需要 /）：
- @router.get("/{id}") → /roles/{id}

特殊端点（不需要 /）：
- @router.get("/mine") → /reservations/mine
- @router.get("/pending/node1") → /reservations/pending/node1

请根据以上规则更新前端拦截器。
```

### ❌ 差的做法

```
/reservations/pending/node1 有斜杠，修复
（修复后）
/reservations/mine 也有斜杠，修复
（修复后）
/museums/{id}/halls 也有斜杠，修复
...
```

## 三、明确任务边界

### ✅ 好的做法

```
请检查以下内容：

1. 后端所有 API 路由定义
2. 前端所有 API 调用
3. 拦截器规则
4. 硬编码的斜杠

输出：
1. 斜杠规则表
2. 需要修复的文件列表
3. 修复后的代码
4. 测试验证报告
```

### ❌ 差的做法

```
确认一下服务状态
```

---

## 四、阶段性确认

### ✅ 好的做法

```
这个任务交给你，请：

1. 全面检查所有相关文件
2. 列出所有潜在问题
3. 一次性修复所有问题
4. 主动测试验证
5. 再向我汇报

不需要每一步都问我，最后统一汇报即可。
```

### ❌ 差的做法

```
修复 A → 确认 → 修复 B → 确认 → 修复 C → 确认
```

## 五、提供可验证的信息

### ✅ 好的做法

```
问题现象：
- 访问 /reservations/mine 返回 []
- 但直接用 curl 测试后端 API 有数据
- Network 标签显示请求 URL 有尾部斜杠

期望结果：
- 返回预约列表数据
- 请求 URL 没有尾部斜杠

测试账号：
- user004@test.com / changeme
```

### ❌ 差的做法

```
不工作，修一下
```

## 七、避免持续纠错

### 问题模式

```
人类：A 有问题
LLM：修复 A
人类：B 也有问题
LLM：修复 B
人类：C 还有问题
...（10+ 轮）
```

### 改进方案

```
人类：请全面检查 A、B、C 相关的所有问题
      一次性列出并修复
      
LLM：已检查，发现以下问题：
      1. A 问题（已修复）
      2. B 问题（已修复）
      3. C 问题（已修复）
      4. D 问题（意外发现，已修复）
      
      已全部测试验证，请确认。
```

---

## 八、给 LLM 的明确要求

### 执行要求

```
请遵循以下原则：

1. 系统性执行
   - 先全面检查相关代码
   - 一次性列出所有潜在问题
   - 批量修复

2. 主动验证
   - 修复后立即测试
   - 确认生效后再提交
   - 提供验证步骤

3. 边界思考
   - 考虑 UUID、数字 ID、嵌套路径
   - 检查硬编码可能性
   - 检查配置文件

4. 清晰同步
   - 提供验证步骤
   - 提供预期结果
   - 告知如何排查
```
