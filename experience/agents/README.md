## 🎯 核心理念

**推荐方案**: **单 Agent + 强制检查清单 + 人类关键节点审查**

**不推荐**: 多 Agent 分工 (同一模型知识边界相同，易"默契"放过问题)

## 🚀 快速开始

### 1. 选择合适的 Prompt 模板

根据任务类型选择：
- 📝 **代码开发** → `prompts/code-development.prompt`
- 🔍 **代码审查** → `prompts/code-review.prompt`
- 🐛 **Bug 修复** → `prompts/bug-fix.prompt`
- ✅ **测试生成** → `prompts/testing.prompt`
- 📊 **任务分析** → `prompts/task-analysis.prompt`

### 2. 提供完整上下文

```markdown
## 任务描述
[清晰描述要做什么]

## 相关文件
- `path/to/file.py`

## 适用 Rules
- [API 斜杠规则](../rules/general/api-slash.rule)
- [代码审查规则](../rules/general/code-review.rule)

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
```

### 3. 指定输出格式

```markdown
请按以下格式输出:

1. 分析
2. 实现
3. 测试
4. 自查报告
```

### 4. 按检查清单验证

使用 [提交前检查清单](../../checklists/before-commit.md) 验证 AI 输出。

## 📝 Prompt 模板

### 代码开发

```prompt
# 角色
你是一名资深 Python 开发工程师，擅长 FastAPI 开发。

# 任务
实现 [功能描述]。

# 要求
1. 遵循 [相关 Rules]
2. 考虑 [边界条件]
3. 添加 [注释/文档]
4. 编写 [单元测试]

# 相关文件
[文件路径]

# 输出格式
1. 分析
2. 实现
3. 测试
4. 自查报告
```

### 代码审查

```prompt
# 角色
你是一名资深代码审查专家，擅长发现潜在问题。

# 任务
审查以下代码变更。

# 变更内容
[描述]

# 影响范围
[模块/功能]

# 审查重点
1. 核心逻辑是否正确
2. 安全性是否有漏洞
3. 性能是否有影响
4. 测试是否充分

# 输出格式
1. 审查结论 (批准/修改/重写)
2. 发现的问题 (按严重程度排序)
3. 改进建议
```

### Bug 修复

```prompt
# 角色
你是一名资深调试专家，擅长定位和修复 Bug。

# 问题描述
[详细描述 Bug 现象]

# 期望行为
[应该是什么样的]

# 实际行为
[实际是什么样的]

# 复现步骤
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

# 相关文件
[文件路径]

# 输出格式
1. 问题分析
2. 根本原因
3. 修复方案
4. 验证方法
```

### 测试生成

```prompt
# 角色
你是一名资深测试工程师，擅长编写高质量单元测试。

# 任务
为 [功能] 编写单元测试。

# 要求
1. 覆盖正常路径
2. 覆盖异常路径
3. 测试边界条件
4. 验证权限控制
5. 使用 Mock 外部依赖

# 相关文件
[文件路径]

# 输出格式
1. 测试文件
2. 测试覆盖说明
3. Mock 说明
```

### 任务分析

```prompt
# 角色
你是一名资深技术负责人，擅长任务拆解和风险评估。

# 任务描述
[详细描述]

# 输出格式
1. 任务理解
2. 技术方案
3. 任务拆解
4. 风险评估
5. 验收标准
```

## 🔄 工作流

### 标准开发工作流

```
1. 任务分析 (人类)
   ↓
2. 编写代码 (AI)
   ↓
3. 编写测试 (AI)
   ↓
4. 运行测试 (AI)
   ↓
5. 按清单自查 (AI)
   ↓
6. 修复问题 (AI)
   ↓
7. 人类审查 (人类)
   ↓
8. 提交代码 (AI)
```

### 详细步骤

详见 [AI 辅助编程最佳实践](../guides/ai-assisted-programming.md#-推荐工作流)

## 💡 最佳实践

### 1. 上下文管理

**✅ 好的做法**:
```markdown
## 项目背景
[项目是做什么的]

## 技术栈
- FastAPI (后端)
- Vue3 (前端)
- PostgreSQL (数据库)
- ElasticSearch (搜索)

## 相关文件
- `app/api/reservations.py`
- `app/services/reservation_service.py`

## 相关 Rules
- [API 斜杠规则](../rules/general/api-slash.rule)
```

**❌ 不好的做法**:
```markdown
帮我写个预约功能。
```

### 2. 任务边界

**✅ 好的做法**:
```markdown
## 任务范围
- 创建预约 API
- 查询预约 API
- 撤回预约 API

## 不在范围
- 审批功能 (另外的任务)
- Excel 导出 (另外的任务)
```

**❌ 不好的做法**:
```markdown
把预约功能都做了。
```

### 3. 阶段性确认

**✅ 好的做法**:
```markdown
请先分析任务，列出实现方案，等我确认后再开始编码。
```

**❌ 不好的做法**:
```markdown
直接开始写代码。
```

### 4. 建设性反馈

**✅ 好的做法**:
```markdown
这个实现有个问题：路由顺序可能导致/admin 被错误匹配。

建议将具体路由 (/admin) 放在参数路由 (/{id}) 之前。

参考：[FastAPI 经验总结](../technical/fastapi-lessons.md)
```

**❌ 不好的做法**:
```markdown
写错了，重写。
```

## ❓ 常见问题

### Q1: AI 写的代码可靠吗？

**A**: 需要验证。

**建议**:
1. ✅ 强制要求测试覆盖
2. ✅ 按检查清单自查
3. ✅ 人类审查关键代码
4. ✅ 持续记录问题

### Q2: 如何让 AI 理解复杂业务？

**A**: 提供完整上下文。

**方法**:
1. 项目背景文档
2. 业务流程图
3. 数据字典
4. 相关 Rules

### Q3: AI 总是忽略某些问题怎么办？

**A**: 更新检查清单。

**方法**:
1. 记录忽略的问题
2. 添加到检查清单
3. 要求 AI 按清单自查
4. 定期回顾更新

### Q4: 如何提高 AI 输出质量？

**A**: 明确期望 + 提供示例。

**方法**:
1. 明确输出格式
2. 提供优秀示例
3. 指出具体问题
4. 持续反馈改进
