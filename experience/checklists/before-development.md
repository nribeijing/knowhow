# 开发前需求确认清单

> 确保在开始编码前充分理解需求，避免返工

**适用场景**: 接收新任务后、开始编码前  
**执行者**: 开发者 (AI + 人类)  
**耗时**: 15-30 分钟

---

## 📋 需求理解

### 1. 问题陈述

- [ ] **能用一句话说清楚要做什么吗？**
  ```
  示例：实现用户撤回待一级审批状态的预约功能
  ```

- [ ] **为什么需要做这个功能？**
  - [ ] 用户需求
  - [ ] Bug 修复
  - [ ] 性能优化
  - [ ] 技术债务
  - [ ] 合规要求

- [ ] **不做会有什么后果？**
  ```
  示例：用户无法撤回错误提交的预约，必须联系管理员删除
  ```

---

### 2. 用户故事

- [ ] **明确目标用户**
  ```
  作为 [角色]，我想要 [功能]，以便 [价值]
  
  示例：作为一般用户，我想要撤回待一级审批的预约，以便修正错误信息后重新提交
  ```

- [ ] **用户使用流程**
  - [ ] 用户从哪个入口进入
  - [ ] 看到什么界面
  - [ ] 执行什么操作
  - [ ] 看到什么反馈
  - [ ] 数据如何变化

---

### 3. 验收标准

- [ ] **功能验收标准 (Functional)**
  ```
  - [ ] 用户可以撤回状态为"待一级审批"的预约
  - [ ] 撤回后预约状态变为"已撤回"
  - [ ] 用户可以重新编辑并提交
  - [ ] 已审批的预约不可撤回
  ```

- [ ] **非功能验收标准 (Non-Functional)**
  ```
  - [ ] 响应时间 < 500ms
  - [ ] 支持并发撤回
  - [ ] 操作记录可追溯
  ```

- [ ] **边界条件**
  ```
  - [ ] 预约已被一级审批通过 → 不可撤回
  - [ ] 预约已被二级审批通过 → 不可撤回
  - [ ] 预约已被取消 → 不可撤回
  - [ ] 非预约用户本人 → 不可撤回
  ```

### 5. API 设计

- [ ] **端点定义**
  ```
  方法：PATCH
  路径：/api/v1/reservations/{id}/withdraw
  权限：预约用户本人
  ```

- [ ] **请求参数**
  ```json
  {
    "reason": "string (可选)"
  }
  ```

- [ ] **响应格式**
  ```json
  {
    "id": "res123",
    "status": "withdrawn",
    "withdrawn_at": "2026-03-16T10:00:00Z"
  }
  ```

- [ ] **错误处理**
  ```
  403 - 无权撤回 (非本人/已审批)
  404 - 预约不存在
  422 - 参数验证失败
  ```

---

### 6. 数据模型

- [ ] **数据库变更**
  ```sql
  -- 是否需要新增字段？
  ALTER TABLE reservations ADD COLUMN withdrawn_at TIMESTAMP;
  ALTER TABLE reservations ADD COLUMN withdraw_reason TEXT;
  ```

- [ ] **状态流转**
  ```
  pending_node1 → withdrawn (允许)
  approved_node1 → withdrawn (不允许)
  approved_node2 → withdrawn (不允许)
  cancelled → withdrawn (不允许)
  ```

- [ ] **数据迁移**
  - [ ] 需要历史数据迁移
  - [ ] 迁移脚本已准备
  - [ ] 回滚方案已准备
