# 设计评审清单

> 系统性审查设计方案，确保技术质量

**适用场景**: Design Doc 评审会议  
**执行者**: 技术负责人 + 团队成员  
**耗时**: 30-60 分钟

---

## 📋 评审前准备

### 文档要求

- [ ] Design Doc 提前 24 小时发送
- [ ] 文档结构完整 (按 design-doc-template.md)
- [ ] 架构图/流程图清晰
- [ ] 替代方案有对比分析

### 评审者要求

- [ ] 至少 2 名评审者
- [ ] 包含技术负责人
- [ ] 评审者提前阅读文档 (30 分钟)
- [ ] 评审者准备问题和意见

---

## 🔍 评审内容

### 1. 问题陈述 (10 分钟)

- [ ] **问题清晰**: 能用一句话说清楚要解决什么问题
- [ ] **目标明确**: 目标符合 SMART 原则
- [ ] **范围合理**: 非目标明确，范围不蔓延
- [ ] **价值明确**: 为什么要做，不做会怎样

**常见问题**:
```
❌ 问题描述模糊 → "提升用户体验"
✅ 问题描述清晰 → "用户无法撤回待审批预约，必须联系管理员"

❌ 目标不可衡量 → "提高性能"
✅ 目标可衡量 → "响应时间从 500ms 降低到 200ms"
```

---

### 2. 技术方案 (20 分钟)

#### 2.1 架构设计

- [ ] **架构图完整**: 组件 + 数据流清晰
- [ ] **职责分离**: 各组件职责单一明确
- [ ] **依赖合理**: 无循环依赖，依赖方向正确
- [ ] **扩展性**: 支持水平/垂直扩展

**评审要点**:
```
- 组件划分是否合理？
- 数据流是否清晰？
- 单点故障在哪里？
- 如何扩展？
```

#### 2.2 数据模型

- [ ] **表结构设计**: 符合范式，无冗余
- [ ] **索引设计**: 查询性能有保证
- [ ] **数据一致性**: 事务边界清晰
- [ ] **数据迁移**: 有迁移方案和回滚计划

**评审要点**:
```
- 表结构是否满足查询需求？
- 索引是否覆盖高频查询？
- 大数据量如何处理？
- 历史数据如何迁移？
```

#### 2.3 API 设计

- [ ] **RESTful 规范**: 符合 API 设计规范
- [ ] **请求/响应**: 示例完整，格式清晰
- [ ] **错误处理**: 错误码和错误消息规范
- [ ] **权限控制**: 认证授权明确

**评审要点**:
```
- API 命名是否规范？
- 参数验证是否完整？
- 错误处理是否统一？
- 权限控制是否到位？
```

---

### 3. 替代方案 (10 分钟)

- [ ] **方案对比**: 至少 2 个替代方案
- [ ] **优缺点分析**: 每个方案有清晰的优缺点
- [ ] **成本估算**: 开发成本/运维成本
- [ ] **选择理由**: 最终选择有充分依据

**评审要点**:
```
- 是否考虑了足够的替代方案？
- 优缺点分析是否客观？
- 成本估算是否合理？
- 选择理由是否充分？
```

---

### 4. 测试策略 (10 分钟)

- [ ] **测试覆盖**: 单元测试/集成测试/E2E 测试
- [ ] **覆盖率目标**: 明确的覆盖率要求 (>80%)
- [ ] **测试数据**: 测试数据准备方案
- [ ] **性能测试**: 性能指标和测试方法

**评审要点**:
```
- 测试覆盖是否充分？
- 覆盖率目标是否合理？
- 如何保证测试质量？
- 性能测试如何做？
```

---

### 5. 监控告警 (5 分钟)

- [ ] **关键指标**: SLO/SLI 明确
- [ ] **告警级别**: P0/P1/P2 定义清晰
- [ ] **仪表盘**: 监控面板设计
- [ ] **响应流程**: 告警响应机制

**评审要点**:
```
- 关键指标是否覆盖核心场景？
- 告警阈值是否合理？
- 如何发现性能问题？
- 告警后如何处理？
```

---

### 6. 风险评估 (5 分钟)

- [ ] **技术风险**: 识别技术难点和风险点
- [ ] **业务风险**: 识别业务影响和用户风险
- [ ] **缓解措施**: 每个风险有应对方案
- [ ] **回滚计划**: 失败后如何回滚

**评审要点**:
```
- 是否识别了主要风险？
- 风险概率和影响评估是否合理？
- 缓解措施是否可行？
- 回滚方案是否完备？
```

---

### 7. 实施计划 (5 分钟)

- [ ] **阶段划分**: 分阶段实施，每阶段有交付物
- [ ] **时间估算**: 时间估算合理，有缓冲
- [ ] **依赖关系**: 识别外部依赖和阻塞项
- [ ] **验收标准**: 每阶段有明确验收标准

**评审要点**:
```
- 阶段划分是否合理？
- 时间估算是否现实？
- 有哪些依赖和阻塞？
- 如何验收？
```

---

## ✅ 评审结论

### 评审结果

- [ ] ✅ **批准** - 无重大问题，可以实施
- [ ] ⚠️ **修改后批准** - 有小问题，修改后无需再审
- [ ] ❌ **需要重审** - 有重大问题，修改后重新评审

### 必须修改的问题

```
1. [问题描述]
   位置：[文档章节]
   建议：[修改建议]
   严重性：[高/中/低]

2. [问题描述]
   ...
```

### 建议改进的问题

```
1. [问题描述]
   建议：[改进建议]
```

## 🔄 评审后跟进

### 作者职责

- [ ] 整理评审意见
- [ ] 修改 Design Doc
- [ ] 回复评审意见 (采纳/不采纳及理由)
- [ ] 更新版本记录

### 评审者职责

- [ ] 确认修改是否到位
- [ ] 批准实施 (如修改后批准)
- [ ] 参与后续评审 (如需要重审)

### 跟进时间线

```
评审后 24 小时内:
- 作者整理意见

评审后 48 小时内:
- 作者完成修改
- 评审者确认

评审后 72 小时内:
- 批准实施/安排重审
```

- ✅ 检查 Design Doc 完整性
- ✅ 按清单预审查

- ✅ 记录评审意见

- ✅ 解答技术问题

- ✅ 整理评审意见
