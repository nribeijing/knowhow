## 🎯 核心原则

### 原则 1: 单 Agent + 检查清单 > 多 Agent

**结论**: 对于中小型项目，**单 Agent + 强制检查清单 + 人类关键节点审查** 是最佳选择。

**理由**:

| 维度 | 单 Agent + 清单 | 多 Agent |
|------|----------------|----------|
| 代码质量 | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 效率 | ⭐⭐⭐⭐ | ⭐⭐ |
| 配置成本 | ⭐⭐⭐⭐ | ⭐ |
| 可维护性 | ⭐⭐⭐⭐ | ⭐⭐ |
| 盲点覆盖 | ⭐⭐⭐⭐ | ⭐⭐⭐ |

**核心洞察**:
> 质量不取决于 Agent 数量，而取决于是否有独立的审查环节和强制的检查清单。

### 原则 2: 人类把关关键节点

**AI 负责**:
- ✅ 编写代码
- ✅ 编写测试
- ✅ 运行测试
- ✅ 按清单自查
- ✅ 修复问题

**人类负责**:
- ✅ 核心逻辑审查
- ✅ 安全性审查
- ✅ 性能影响评估
- ✅ 批准提交

### 原则 3: 测试驱动，Mock 隔离

**测试策略**:
```
API 层测试 → 用测试 DB (验证 SQL)
服务层测试 → Mock 外部依赖 (ES/Redis)
纯逻辑测试 → 无需 DB (纯验证)
```

**Mock 原则**:
- ✅ Mock 外部依赖 (ES、Redis、第三方 API)
- ✅ 使用测试容器 (PostgreSQL、Redis)
- ✅ 测试隔离 (每个测试独立状态)

## 🔄 推荐工作流

### 标准开发流程

```
1. 任务分析 (人类)
   ↓
2. 编写代码 (AI)
   ↓
3. 编写测试 (AI)
   ↓
4. 运行测试 (AI)
   ↓
5. 按清单自查 (AI)
   ↓
6. 修复问题 (AI)
   ↓
7. 人类审查 (人类)
   ↓
8. 提交代码 (AI)
```

### 详细步骤

#### 1. 任务分析 (人类)

```markdown
## 任务描述

**目标**: [清晰描述要做什么]

**验收标准**:
- [ ] 标准 1
- [ ] 标准 2

**相关文件**:
- `path/to/file.py`

**适用 Rules**:
- [API 斜杠规则](../../../knowledge/rules/api-slash.rule)
- [代码审查规则](../../../knowledge/rules/code-review.rule)
```

#### 2. 编写代码 (AI)

**AI 职责**:
- 遵循项目规范
- 编写完整实现
- 添加必要注释
- 考虑边界条件

**示例 Prompt**:
```
请实现 [功能描述]。

要求:
1. 遵循 [相关 Rules]
2. 考虑 [边界条件]
3. 添加 [注释/文档]

相关文件:
[文件路径]
```

#### 3. 编写测试 (AI)

**测试覆盖**:
- ✅ 正常路径
- ✅ 异常路径
- ✅ 边界条件
- ✅ 权限验证

**示例 Prompt**:
```
请为 [功能] 编写单元测试。

要求:
1. 覆盖正常/异常路径
2. 测试边界条件
3. 验证权限控制
4. 使用 Mock 外部依赖
```

#### 4. 运行测试 (AI)

**AI 职责**:
- 运行所有相关测试
- 记录测试结果
- 分析失败原因
- 提出修复方案

#### 5. 按清单自查 (AI)

**使用检查清单**:
- [提交前检查清单](../../../checklists/before-commit.md)
- [代码审查清单](../../../checklists/code-review.md)

**AI 自查 Prompt**:
```
请按照检查清单自查代码:

1. 功能正确性
   - [ ] 所有测试通过
   - [ ] 边界条件已测试
   - [ ] 错误处理完整

2. 代码质量
   - [ ] 遵循代码规范
   - [ ] 无重复代码
   - [ ] 注释清晰

3. 安全性
   - [ ] 输入验证完整
   - [ ] 权限检查到位
   - [ ] 无敏感信息

请逐项检查并报告结果。
```

#### 6. 修复问题 (AI)

**AI 职责**:
- 修复所有发现的问题
- 重新运行测试
- 更新文档
- 再次自查

#### 7. 人类审查 (人类)

**人类审查重点**:
- 🔍 核心业务逻辑
- 🔍 安全性相关代码
- 🔍 性能关键路径
- 🔍 外部接口调用

**审查 Prompt**:
```
请审查以下代码变更:

**变更内容**: [描述]
**影响范围**: [模块/功能]
**测试覆盖**: [测试数/通过率]

重点关注:
1. 核心逻辑是否正确
2. 安全性是否有漏洞
3. 性能是否有影响
4. 测试是否充分
```

#### 8. 提交代码 (AI)

**AI 职责**:
- 编写规范的提交信息
- 更新相关文档
- 推送到远程仓库
- 记录变更日志

## 💡 实战经验

> ⚠️ **注意**: 5 个核心陷阱的详细说明已移至 [常见陷阱](./common-pitfalls.md)，此处仅保留要点。

### 经验 1: 路由顺序陷阱

**要点**: FastAPI 按定义顺序匹配路由，静态路由必须放在参数路由之前。

**详见**: [常见陷阱 - 路由顺序](./common-pitfalls.md#1️⃣-路由顺序陷阱) | [案例](../../case-studies/api-route-order-issue.md)

---

### 经验 2: Mock 异步方法

**要点**: AsyncMock 的属性也是 Mock，不能直接 await，需用异步函数包装。

**详见**: [常见陷阱 - Mock 异步](./common-pitfalls.md#2️⃣-mock-异步方法陷阱) | [案例](../../case-studies/mock-async-method-issue.md)

---

### 经验 3: 权限验证

**要点**: 使用正确的依赖注入（如 `get_current_admin_user` 而非 `get_current_user`）。

**详见**: [常见陷阱 - 权限验证](./common-pitfalls.md#3️⃣-权限验证陷阱) | [案例](../../case-studies/permission-validation-issue.md)

---

### 经验 4: 测试隔离

**要点**: 全局状态导致测试相互影响，使用 autouse fixture 自动清理。

**详见**: [常见陷阱 - 测试隔离](./common-pitfalls.md#4️⃣-测试隔离陷阱) | [案例](../../case-studies/test-isolation-issue.md)

---

### 经验 5: 响应模型验证

**要点**: FastAPI 严格验证响应模型，使用 Helper 函数生成完整数据。

**详见**: [常见陷阱 - 响应验证](./common-pitfalls.md#5️⃣-响应验证陷阱) | [案例](../../case-studies/response-validation-issue.md)

## ❓ 常见问题

### Q1: 应该使用多 Agent 吗？

**A**: 对于中小型项目，**不推荐**。

**理由**:
1. 同一模型的知识边界相同
2. Agent 之间可能"默契"放过问题
3. 上下文切换成本高
4. 配置复杂度高

**推荐**: 单 Agent + 检查清单 + 人类审查

---

### Q2: 如何确保 AI 代码质量？

**A**: 三层保障：

1. **检查清单** - AI 按清单自查
2. **测试覆盖** - 强制要求测试通过
3. **人类审查** - 关键节点人类把关

---

### Q3: AI 写的测试可靠吗？

**A**: 需要验证：

1. ✅ 测试是否真正运行
2. ✅ 覆盖率是否充分
3. ✅ Mock 是否正确
4. ✅ 边界条件是否测试

**建议**: 人类抽查测试代码

---

### Q4: 如何处理 AI 的错误？

**A**: 建立错误记录机制：

1. 记录错误类型
2. 分析根本原因
3. 更新检查清单
4. 避免重复犯错

**示例**:
```markdown
## 错误记录

**日期**: 2026-03-15
**问题**: 路由顺序导致 404
**原因**: AI 忽略了 FastAPI 路由匹配规则
**解决**: 更新检查清单，添加"路由顺序"项
**预防**: 所有路由定义后 AI 自查顺序
```

## ✅ 检查清单

### 提交前检查清单

```markdown
## 功能正确性
- [ ] 所有测试通过
- [ ] 边界条件已测试
- [ ] 错误处理完整

## 代码质量
- [ ] 遵循代码规范
- [ ] 无重复代码
- [ ] 注释清晰
- [ ] 路由顺序正确 (具体在前，参数在后)

## 安全性
- [ ] 输入验证完整
- [ ] 权限检查到位 (使用正确的依赖注入)
- [ ] 无敏感信息泄露

## 测试质量
- [ ] Mock 正确 (异步方法用函数包装)
- [ ] 测试隔离 (无全局状态污染)
- [ ] 覆盖率充分

## 响应验证
- [ ] 返回数据包含所有必填字段
- [ ] 类型正确
- [ ] 无 None 值 (除非 Optional)

## 文档
- [ ] 更新 README
- [ ] 更新 API 文档
- [ ] 提交信息规范
```

### 代码审查清单

```markdown
## 审查重点

### 核心逻辑
- [ ] 业务逻辑正确
- [ ] 边界条件处理
- [ ] 异常处理完整

### 安全性
- [ ] 权限验证正确
- [ ] 输入验证完整
- [ ] SQL/NoSQL 注入防护

### 性能
- [ ] 无 N+1 查询
- [ ] 索引使用合理
- [ ] 缓存策略适当

### 测试
- [ ] 测试覆盖充分
- [ ] Mock 正确
- [ ] 测试隔离

## 审查结论

- [ ] 批准提交
- [ ] 需要修改 (说明原因)
- [ ] 需要讨论 (列出问题)
```
