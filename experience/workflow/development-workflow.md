## 🎯 核心原则

### 1. 流程是保护，不是束缚

**目的**：
- ✅ 预防问题，而不是事后救火
- ✅ 保证质量，而不是拖延进度
- ✅ 帮助 AI 执行，而不是限制 AI

**理念**：
> 好的流程让正确的事情变得容易，错误的事情变得困难。

---

### 2. 渐进式改进

**不追求一步到位**，而是：

```
当前状态 → 过渡态 → 理想状态

过渡态特点：
- 保留核心质量门禁
- 简化文档负担
- 适合小团队快速迭代
```

**优先级**：
| 优先级 | 环节 | 实施时间 |
|--------|------|----------|
| 🔴 P0 | 需求确认 + 设计评审 | 本周 |
| 🟡 P1 | 单元测试 + 自测 | 本月 |
| 🟢 P2 | 监控 + 复盘 | 下季度 |

## 📋 流程详解

### 阶段 1: 需求分析 (Requirement)

**目标**: 确保理解正确，避免返工

**理想流程**:
```
输入：产品需求/用户反馈
  ↓
需求评审 (Requirement Review)
  ↓
编写测试用例 (Test Case)
  ↓
输出：清晰的需求文档 + 验收标准
```

**当前实践**:
```
输入：产品需求
  ↓
使用 before-development.md 清单确认
  ↓
输出：任务描述 + 验收标准
```

**AI 执行指南**:
```markdown
## AI 在这个阶段的任务

1. 按清单确认需求
   → 使用 experience/checklists/before-development.md

2. 输出需求理解
   → 问题陈述 (1 句话)
   → 用户故事 (作为...想要...以便...)
   → 验收标准 (可验证的列表)

3. 识别风险
   → 技术难点
   → 依赖项
   → 边界条件
```

- [ ] [开发前需求确认清单](../checklists/before-development.md)
- [ ] [任务描述模板](../templates/task-description-template.md)

### 阶段 2: 设计 (Design)

**目标**: 确保技术方案合理，避免架构问题

**理想流程**:
```
输入：需求文档
  ↓
编写 Design Doc
  ↓
设计评审 (Design Review)
  ↓
API Spec Review
  ↓
输出：设计文档 + API 规范
```

**当前实践**:
```
输入：需求文档
  ↓
核心功能编写 Design Doc
  ↓
输出：技术方案 + API 设计
```

**AI 执行指南**:
```markdown
## AI 在这个阶段的任务

1. 技术方案设计
   → 架构图 (组件 + 数据流)
   → 数据模型 (表结构 + 关系)
   → 替代方案分析 (至少 2 个方案)

2. API 设计
   → 端点定义 (方法 + 路径)
   → 请求/响应示例
   → 错误处理

3. 风险评估
   → 技术难点
   → 性能影响
   → 安全风险
```

### 阶段 3: 开发 (Develop)

**目标**: 高质量代码实现

**理想流程**:
```
输入：设计文档 + API 规范
  ↓
编码实现
  ↓
单元测试 (Unit Test)
  ↓
自测 (Self-Test)
  ↓
代码审查 (Code Review)
  ↓
输出：可工作的代码 + 测试
```

**当前实践**:
```
输入：设计文档
  ↓
编码实现
  ↓
代码审查 (Code Review)
  ↓
输出：代码 + 测试
```

**AI 执行指南**:
```markdown
## AI 在这个阶段的任务

1. 编码实现
   → 遵循代码规范
   → 添加必要注释
   → 考虑边界条件

2. 编写测试
   → 单元测试 (覆盖率 >80%)
   → 集成测试 (关键路径)
   → Mock 外部依赖

3. 按清单自查
   → 功能正确性
   → 代码质量
   → 安全性
   → 测试质量

4. 准备审查
   → 编写 PR 描述
   → 标注审查重点
   → 提供测试说明
```

### 阶段 4: 测试 (Testing)

**目标**: 发现并修复问题

**理想流程**:
```
输入：代码 + 测试
  ↓
部署到 QA 环境
  ↓
QA 测试
  ↓
部署到生产环境
  ↓
生产验证测试
  ↓
输出：测试报告 + 发布说明
```

**当前实践**:
```
输入：代码 + 测试
  ↓
部署到生产环境
  ↓
生产验证测试
  ↓
输出：发布说明
```

**AI 执行指南**:
```markdown
## AI 在这个阶段的任务

1. 准备测试环境
   → 测试数据准备
   → 环境配置检查

2. 执行测试
   → 运行自动化测试
   → 记录测试结果
   → 分析失败原因

3. 生产验证
   → 核心功能验证
   → 监控指标检查
```

### 阶段 5: 运维 (Ops)

**目标**: 稳定运行 + 持续改进

**理想流程**:
```
输入：发布的应用
  ↓
监控指标
  ↓
事件响应 (Incident)
  ↓
事故复盘 (Postmortem)
  ↓
输出：监控报告 + 改进措施
```

**当前实践**:
```
输入：发布的应用
  ↓
基础监控
  ↓
问题响应
```

**AI 执行指南**:
```markdown
## AI 在这个阶段的任务

1. 监控配置
   → 关键指标定义
   → 告警阈值设置

2. 问题响应
   → 日志分析
   → 根因定位
   → 修复方案

3. 事故复盘
   → 记录时间线
   → 分析根本原因
   → 提出改进措施
```

## 🤖 AI 角色定位

### AI 是流程执行者，不是流程制定者

**AI 的职责**:
- ✅ 按流程执行每个阶段
- ✅ 使用模板输出标准化文档
- ✅ 按清单自查质量
- ✅ 发现流程问题并反馈

**人类的职责**:
- ✅ 制定和优化流程
- ✅ 关键节点评审 (需求/设计/审查)
- ✅ 处理异常情况
- ✅ 持续改进流程

- [ ] 按清单完成自查

## 🔄 流程改进

### 如何更新本流程？

1. **发现问题** → 记录到流程问题清单
2. **分析原因** → 是流程缺失还是执行不到位？
3. **提出方案** → 新增环节/简化环节/优化环节？
4. **团队评审** → 讨论通过后更新
5. **更新文档** → 修改本流程 + 相关模板/清单

### 改进原则

- ✅ 流程服务于目标，不是目标本身
- ✅ 简化能简化的环节，强化必须强化的环节
- ✅ 每次改进都要有明确的问题驱动
- ✅ 改进后观察效果，持续调整
